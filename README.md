# Issues AI Analyzer 🤖📊

An automated tool for analyzing GitHub issues and generating structured, decision-ready reports for R&D Program Management and Agile Transformation teams using Anthropic Claude AI.

## 🚀 How It Works
1. The script connects to the GitHub API to fetch recent open or closed issues based on the specified repository.
2. It parses and cleans the raw JSON data, extracting essential metadata such as labels, assignees, and descriptions.
3. The processed data is sent to the Anthropic Claude API along with advanced system prompts enforcing a strict data contract (no hallucinations, mandatory traceability).
4. The AI generates a comprehensive, business-ready report in Markdown format.
5. The final output is automatically saved into the `reports/` directory.

## ⚙️ Setup (Installation & Configuration)

To run the project locally, make sure you have Python version 3.8 or newer installed.

1. Clone the repository:
   git clone [https://github.com/your-username/issues-ai-analyzer.git](https://github.com/your-username/issues-ai-analyzer.git)
   cd issues-ai-analyzer

2. Create and activate a virtual environment:
   python -m venv venv

   **Windows:**
   venv\Scripts\activate

   **Linux/macOS:**
   source venv/bin/activate

3. Install the required libraries:
   pip install -r requirements.txt

4. Configure the .env file:
   Create a .env file in the root folder to store configuration, using .env.example as a template.
   Make sure to configure your API keys and repository details:
   
   ANTHROPIC_API_KEY=your_api_key
   GITHUB_OWNER=repository_owner
   GITHUB_REPO=repository_name
   GITHUB_ISSUES_LIMIT=100
   CLAUDE_MODEL=claude-3-opus-20240229
   CLAUDE_MAX_TOKENS=4000

## 💻 Usage
To run the analysis and generate reports, simply execute the main script:

   python main.py

The generated reports will be saved in the newly created `reports/` directory as .md files.

## 🤖 Automation & AI Learnings

- **Issues #25, #28, #40, and #63** all involved manually propagating a schema/contract change across mock generators, tests, and data loaders → a GenAI-assisted "contract change impact" step (LLM scanning the repo for references to changed JSON-schema fields and proposing diffs) integrated into PR review on `schemas/*` edits would have surfaced the downstream impact set up front and removed at least one rework cycle per change.
- **Issues #101 and #59** both involved configuration drift between code, `.env.example`, and `docker-compose.yml` that was caught reactively by humans → a deterministic CI linter (no AI needed) plus an LLM-based "secrets and config review" check on PRs touching `docker-compose*.yml` or `.env*` would have caught the hardcoded credential and missing env keys before merge, eliminating the discovery-as-bug pattern entirely.
